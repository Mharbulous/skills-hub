# Skill Updater

Apply a known change to an existing Skills-hub source skill. The user has
already decided what to change; this skill's job is to make that edit cleanly in
the canonical source tree.

## Source of Truth

For this repository, the editable source is:

```text
public/skills/<skill-name>/
```

Generated harness outputs under `public/claude/`, `public/codex/`,
`public/cowork/`, Cowork `.skill` packages, runtime plugin caches, and product
repo wrapper copies are not source. Do not edit them as the delivery path.

If the user asks to inventory, install, update, replace stale wrappers, import
Cowork packages, or manage resolver wrappers, stop and use `skills-hub` instead.

## Workflow

1. Identify the target skill in `public/skills/<skill-name>/`. If the user gave
   an explicit path, verify it is inside `public/skills/` before editing.
2. Read the target file before editing. For a description or trigger change,
   edit the canonical `SKILL.md` frontmatter. For instruction changes, edit the
   relevant source file under the same skill directory.
3. Keep the edit surgical. Do not rewrite unrelated files, run package builders,
   or change generated harness output unless the user explicitly asks for a
   build/deploy follow-up.
4. If changing skill delivery, bootstrap, packaging, resolver behavior, or
   Cowork installation instructions, read `VISION.md` first and ensure the edit
   preserves signed Skills-hub artifacts as the distribution source of truth.
5. After editing, report the changed file paths and the verification performed.

## Review Requests

If the user wants to review before the change lands, make the edit in the repo
and point them to `git diff`. Do not package a preview `.skill` as the review
mechanism for Skills-hub source changes; Cowork packages are generated by
`build/build_index.py` from verified source.

## What Not To Do

- Do not create or update product-repo wrapper/source split layouts.
- Do not package skills from a product repository.
- Do not edit installed runtime cache copies as the documented delivery path.
- Do not fetch remote `SKILL.md` content and follow it before local
  verification.
- Do not run evals automatically. Offer them as a separate
  `skill-creator-improved` step if relevant.
