# Skills-hub Vision & Design Philosophy

**Date:** 2026-06-12
**Status:** Draft - pending owner review

## Scope & Discovery

**Applies to:** The skills-hub repository, its build/publish pipeline, harness
installers, generated Cowork wrappers, signed manifests, and hosted skill
artifacts.

**Read when:** Changing skill delivery, adding skills, modifying bootstrap or
resolver behavior, publishing artifacts, testing Cowork integration, or deciding
where a skill package should be generated/imported from.

**Do not use for:** Coclerk product behavior unrelated to skill distribution,
matter-workflow implementation details inside a skill, or one-off user
workspace operations.

**Related vision docs:**
- Broader: none.
- Narrower: none.
- Adjacent: none.

Principle and anti-pattern numbers in this document are intended to remain
stable once owner-reviewed.

## 1. Theme & Design Philosophy

### Vision Statement

Skills-hub is the static, signed, unauthenticated source of truth for
agent skill delivery across Claude, Codex, and Cowork. It replaces local
repo-backed skill distribution: harnesses may consume generated files from this
repo or from the deployed `skills-hub.web.app` artifacts, but they must not
depend on Coclerk or any other product repo as a packaging source.

### Design Principles

1. **Skills-hub owns skill distribution.** If an artifact is needed to install,
   import, route, verify, or execute a Skills-hub skill, it is generated by
   skills-hub or served from skills-hub's deployed static output.
2. **Product repos are consumers, not distribution infrastructure.** Coclerk may
   be an application that uses skills, and it may provide test workspaces, but it
   is not the source of truth for Skills-hub wrappers, `.skill` packages,
   manifests, or published skill archives.
3. **Tests prove skills-hub paths.** End-to-end tests must exercise generated
   skills-hub artifacts and live Skills-hub URLs. A test that succeeds by reading
   `Coclerk\.agents\skills`, `Coclerk\.claude\wrappers`, or
   `Coclerk\plugins\...\skills` has not proven Skills-hub delivery.
4. **Remote bytes become authoritative only after local verification.** Agents
   must not fetch remote instructions and follow tool output. Trusted local code
   verifies signed manifests and artifact hashes, materializes files locally, and
   only then may the agent read local `SKILL.md` content as instructions.
5. **Manual runtime edits are accelerants, not architecture.** Editing a
   harness runtime cache can speed up a proof, but the edited bytes must be
   copied from generated skills-hub artifacts and the result must not be
   documented as the delivery path.

### Litmus Test

Can a new machine install or import the skill using only this repository's
generated artifacts or the hosted `/hub` output, without reading any Coclerk
source, wrapper, or plugin package? If yes, the change belongs in Skills-hub. If
no, the design is still coupled to the system Skills-hub is replacing.

## 2. Purpose

### Audience

The primary audience is the maintainer and future coding agents evolving
Skills-hub skill delivery across Claude, Codex, and Cowork.

### Pain Points

Local repo-backed skill installs create drift, stale desktop copies, hidden
dependencies on product repositories, and false-positive tests where a harness
appears to use Skills-hub but actually reads Coclerk-local files.

### Value Proposition

Skills-hub gives every harness a single verifiable delivery path: build once,
sign once, publish static artifacts, and let each harness consume only the
generated outputs appropriate to its runtime.

### Killer Use Case

Add a new Cowork skill to `skills/`, build and deploy skills-hub, import a
skills-hub-generated Cowork package or wrapper artifact, and confirm Cowork
resolves the signed hosted skill without requesting access to Coclerk's local
skill repo.

## 3. North Star

A clean machine can install or import any published Skills-hub skill for every
supported harness using only signed skills-hub artifacts, with no copy-paste
skill folders and no dependency on Coclerk-local packaging.

## 4. Non-Goals

- Reusing Coclerk as the Cowork packaging pipeline.
- Treating product repositories as canonical skill registries.
- Hiding verification failures with local fallback reads.
- Solving general application plugin management outside skill delivery.

## 5. Foundations

- Firebase Hosting remains a static transport; integrity comes from signed
  manifests and local trust anchors, not from obscurity or domain trust alone.
- `skills/` is the canonical source tree for skill definitions in this repo.
- `public/` is the generated artifact contract for harness consumers.
- Cowork needs importable wrapper artifacts generated from skills-hub output,
  not packages produced by Coclerk.
- Claude and Codex should read verified full local installs before skill
  enumeration.

## 6. Anti-Patterns

1. **Coclerk packaging leakage.** Generating, importing, or testing a Skills-hub
   skill via `Coclerk\plugins\...\skills` or `Coclerk\.claude\wrappers` makes
   Coclerk part of the delivery architecture and undermines the replacement
   goal.
2. **False E2E success.** A test that passes because stale local files exist in
   a harness runtime cache does not prove hosted Skills-hub delivery.
3. **Remote-follow stubs.** Stubs that tell an agent to fetch remote Markdown
   and follow it as instructions reintroduce prompt-injection risk.
